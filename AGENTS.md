1. 产品默认值和与部署无关的常量写在代码里，不要为每个小细节增加环境变量。秘密、凭据和实例相关配置必须通过环境变量或 Secret 注入，禁止硬编码到仓库。

2. 不要为了兼容删的时候不删干净, 默认按照无保留的删, 为了兼容就保留这么多垃圾是不对的

3. 本地如果测试环境缺失可以使用docker补全, 测试时临时启动的服务/容器/网络/卷等资源命名必须使用 `tmp` 前缀, 测试完成之后只清理这些 `tmp` 前缀资源, 不要动没有 `tmp` 前缀且你不知道用途的东西

4. 不要因为你假想的用户来给他们写一些辅助功能, 特别是在/admin界面下, 管理员知道应该怎么干

5. 垃圾的东西不需要测试, 这个项目只需要测试核心语义, 以及一条产品链即可, 尤其是不可能回归的垃圾问题不要测试, 比如之前写了一个同一个组内只能有一种类型的, 多种类型就报错, 然后后来把这个功能删掉了, 补了一个多种类型不报错的测试, 这测试显然一点意义都没有, 放在这里就是污染代码

6. 发布新版本前需要手动更新 chart 中的版本号，然后创建 tag 并推送。workflow 会自动构建镜像并推送到 Docker Hub，但 chart 版本号需要手动维护。检查清单：
   - [ ] 更新 charts/revlm/values.yaml 的 image.tag 字段
   - [ ] 更新 charts/revlm/Chart.yaml 的 appVersion 字段（必须与 image.tag 一致）
   - [ ] 提交变更到 master 分支
   - [ ] 创建新的版本 tag（例如 `git tag v0.25.2`）
   - [ ] 推送 tag 到 GitHub（`git push origin v0.25.2`）
   - [ ] 确认 Docker Hub 上新镜像成功推送

7. 你需要及时提交你的更改, 做到更改最小化, 每完成一个任务提交一次, 提交产物必须完整可编译, 这样可以很好的方便reviewer审查代码

8. 你的提交需要使用commit, 之后不要push, 尽可能的少push, 只有当你认为完成了所有任务之后再把所有更改push上去

9. 关于性能方面, 你需要严格的控制性能, 这个项目性能作为很重要的评估因素, 一个性能优化合格的标准为: 优化了>10%(包括cpu, 内存, 存储), 5ms以内作为硬件噪音忽略, 如果优化了但是不多, 但是代码行数少掉了也被视为一个很好的优化, 但是如果复杂度又高了且代码行数又多了, 那这就是一个糟糕的代码, 可以毫不犹豫的把他扔掉

10. 对于功能性提交, 你需要严格控制代码行数, 比如一个数据库的聚合查询, 本质就是一个分块, 300行以内都算多的那种, 绝对不能把他写到1000行

11. 关于测试, 测试的内容应该为核心语义, 与高概率回归问题, 包括用户的支付流程, 数据库用量查询流程等, 很多时候一个新增功能是不需要补测试的, 你可能只需要临时来一个测试, 通过之后就可以把测试删掉了, 比如数据库查询的聚合是假的, 时间复杂度高了一个量级, 这种是非常严重的问题, 你可以补一个大型性能测试来验证并且防止回归, 但是比如兑换码的兑换有效性, 这个问题显然几乎不可能回归, 脑子想破了都不知道这个问题怎么回归, 这种就是垃圾测试, 直接扔掉, 最多刚写完成之后测试一下你写的代码怎么样起到一个review的作用, 注:大型性能测试不应该融入项目主测试逻辑中, 需要单独作为脚本拿出来
